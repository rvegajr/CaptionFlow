## 1. Engineering principles â€” what TDD and ISP mean here, concretely

### 1.1 TDD as practiced in this project

The cycle is **Red â†’ Green â†’ Refactor**. Always.

- **Red.** Before writing production code, write the smallest test that fails for the right reason. "Right reason" means: it fails because the production code is missing or incomplete, not because of a typo or missing import.
- **Green.** Write the minimum code that makes the test pass. Resist generalizing. If a test only requires `return 2`, that is what you write. Generalization happens under pressure from the *next* test.
- **Refactor.** With tests green, improve the code's structure without changing behavior. Run the tests after every change. If you can't refactor in confidence, the test suite is too coarse.

**What TDD does not mean here:**

- It does not mean "write tests for everything that exists." It means tests *drive* the existence of code.
- It does not mean 100% line coverage as a target. Coverage is a diagnostic, not a goal.
- It does not mean every test is a unit test. Integration and E2E tests can be the test that drives a design â€” they're slower, so we use them where unit tests can't reach (database queries, end-to-end LTI flow).

**What we do not test:**

- Drizzle's own behavior. We trust the library.
- Fastify's routing. We trust the library.
- Third-party API responses, except via recorded fixtures.

### 1.2 ISP as practiced in this project

A consumer should depend only on the methods it uses. We achieve this by writing **small, role-named interfaces**, and having a single concrete class implement several of them.

**Example â€” the wrong way (one fat interface):**

```ts
interface CaptionService {
  create(...): Caption;
  update(...): Caption;
  delete(...): void;
  getById(...): Caption;
  listByInstructor(...): Caption[];
  search(...): Caption[];
  countViews(...): number;
}
```

A handler that just reads a caption ends up depending on `delete`, `update`, `countViews`, etc., which makes its tests bloated and its responsibilities unclear.

**Example â€” the right way (segregated interfaces):**

```ts
interface CaptionReader {
  getById(id: string): Promise<Caption | null>;
  listByInstructor(instructorId: string): Promise<Caption[]>;
}

interface CaptionWriter {
  create(input: NewCaption): Promise<Caption>;
  update(id: string, patch: Partial<NewCaption>): Promise<Caption>;
  delete(id: string): Promise<void>;
}

interface CaptionSearcher {
  search(query: string): Promise<Caption[]>;
}

class PgCaptionRepo implements CaptionReader, CaptionWriter, CaptionSearcher {
  // one implementation, three roles
}
```

The route handler for `GET /api/captions/:id` accepts `CaptionReader`. Its test fake is two methods, not seven.

**Naming convention:** role-named interfaces (`CaptionReader`, `MagicLinkSender`, `Clock`) are nouns describing capability. The implementation gets a prefix indicating its substrate (`PgCaptionRepo`, `ResendMagicLinkSender`, `SystemClock`).

### 1.3 The interaction between TDD and ISP

ISP makes TDD cheap. When the consumer depends on a two-method interface, the test fake is a two-method object literal, not a mocking framework call. **We do not use `jest.mock` or `vi.mock` for our own modules.** We pass plain object fakes that implement the small interface. This rule is what keeps the test suite readable five years from now.

**Two structural rules embedded in this tree, worth being explicit about:**

1. **`src/shared` is platform-neutral.** Anything in there must run in the browser and in Node, with no DOM and no Node APIs. The cue parser lives here precisely so the same code validates uploads on the server *and* parses files on the client.
2. **`*.test.ts` lives next to its subject by default.** Cross-cutting integration and E2E tests live under `tests/`. This is the only test-location split.

```ts
  id: uuid('id').primaryKey().defaultRandom(),

  email: text('email').notNull(),
  displayName: text('display_name'),

  emailIdx: uniqueIndex('instructor_email_unique').on(t.email),
  ltiIdx: index('instructor_lti').on(t.institutionId, t.ltiSub),
}));

export const caption = pgTable('caption', {
  id: uuid('id').primaryKey().defaultRandom(),
  instructorId: uuid('instructor_id').notNull().references(() => instructor.id),
  youtubeVideoId: text('youtube_video_id').notNull(),
  format: text('format', { enum: ['srt', 'vtt'] }).notNull(),
  languageCode: text('language_code').notNull(),
  contentText: text('content_text').notNull(),
  isMachineTranslated: boolean('is_machine_translated').default(false).notNull(),
  sourceCaptionId: uuid('source_caption_id'),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).defaultNow().notNull(),
}, (t) => ({
  instructorVideoIdx: index('caption_instructor_video').on(t.instructorId, t.youtubeVideoId),
}));

export const captionedResource = pgTable('captioned_resource', {
```

### 7.3 Migration discipline

- `npm run db:generate` after every schema change. Generated SQL is committed to the repo. **Hand-edit migrations only when adding non-schema operations** (data backfills, index renames). Never edit a generated migration's auto-generated SQL.
- `npm run db:migrate` runs at app start in production via the Railway pre-deploy hook (see Â§18).
- For destructive changes (column removal, type changes), generate **two** migrations: one expand, one contract, deployed in two releases. This is overkill for Phase 1 with no real users, but it's the muscle memory we want from the start.

### 7.4 Pool factory

```ts

  CaptionReader, CaptionWriter, Caption, NewCaption, CaptionFormat
```

### 10.3 Green â€” minimum implementation

```ts
// src/shared/srt.ts
import { ok, err, type Result } from './result.js';
import type { Cue } from './cues.js';

export function parseSrt(input: string): Result<Cue[], string> {
  const lines = input.replace(/\r\n/g, '\n').split('\n');
  const cues: Cue[] = [];
  let i = 0;
  while (i < lines.length) {
    if (lines[i]?.trim() === '') { i++; continue; }
    i++; // skip index
    const timing = lines[i++];
    if (!timing) return err('missing timing');
    const m = timing.match(/(\d{2}):(\d{2}):(\d{2}),(\d{3}) --> (\d{2}):(\d{2}):(\d{2}),(\d{3})/);
    if (!m) return err(`bad timing: ${timing}`);
    const toSec = (h: string, mn: string, s: string, ms: string) =>
      Number(h) * 3600 + Number(mn) * 60 + Number(s) + Number(ms) / 1000;
    const startSec = toSec(m[1]!, m[2]!, m[3]!, m[4]!);
    const endSec   = toSec(m[5]!, m[6]!, m[7]!, m[8]!);
    const textLines: string[] = [];
    while (i < lines.length && lines[i]?.trim() !== '') textLines.push(lines[i++]!);
    cues.push({ startSec, endSec, text: textLines.join('\n') });
  }
  return ok(cues);
}
```

Test passes. Green.

### 10.4 Drive the rest with one test at a time

The next tests (write each in turn, watch fail, write minimum, watch pass):

1. Multiple cues separated by blank lines.
2. CRLF line endings (`\r\n`).
3. Multi-line cue text.
4. UTF-8 BOM at start of file (strip it).
5. Cue index lines that aren't sequential (some authoring tools).
6. Returns `err` on malformed timing.
7. Returns `err` if `endSec < startSec`.

```ts
export class FakePlayer implements YouTubePlayer {

  private s: PlayerState = 'idle';

  getState() { return this.s; }

  setTime(t: number) { this.t = t; }
  setState(s: PlayerState) { this.s = s; this.subs.forEach((cb) => cb(s)); }
}

  private pending = new Map<number, (ts: number) => void>();
  private now = 0;
  schedule(cb: (ts: number) => void) { const h = this.next++; this.pending.set(h, cb); return h; }
  cancel(h: number) { this.pending.delete(h); }
  tick(deltaMs = 16) {
```

### 11.5 Green â€” the implementation

```ts
// src/web/caption-surface/sync-engine.ts
export function makeSyncEngine(deps: SyncEngineDeps): SyncEngine {
  const { player, raf, cues, onActiveCueChange } = deps;
  let activeIndex: number | null = null;
  let rafHandle: number | null = null;
  let unsub: (() => void) | null = null;
  let stableFramesSinceBuffering = Infinity; // see Â§11.7

  const findCueIndex = (t: number): number | null => {
    let lo = 0, hi = cues.length - 1;
    while (lo <= hi) {
      const mid = (lo + hi) >> 1;
      const c = cues[mid]!;
      if (t < c.startSec) hi = mid - 1;
      else if (t >= c.endSec) lo = mid + 1;
      else return mid;
    }
    return null;
  };

  const tick = () => {
    if (player.getState() !== 'playing') { rafHandle = null; return; }
    if (stableFramesSinceBuffering < 2) {
      stableFramesSinceBuffering++;
      rafHandle = raf.schedule(tick);
      return;
    }
    const idx = findCueIndex(player.getCurrentTime());
    if (idx !== activeIndex) { activeIndex = idx; onActiveCueChange(idx); }
    rafHandle = raf.schedule(tick);
  };

  const onState = (s: PlayerState) => {
    if (s === 'playing') {
      stableFramesSinceBuffering = 0;
      if (rafHandle === null) rafHandle = raf.schedule(tick);
    } else if (s === 'buffering') {
      stableFramesSinceBuffering = 0;
    } else {
      // paused / ended / idle: resolve once at the current time, then stop the loop
      const idx = findCueIndex(player.getCurrentTime());
      if (idx !== activeIndex) { activeIndex = idx; onActiveCueChange(idx); }
      if (rafHandle !== null) { raf.cancel(rafHandle); rafHandle = null; }
    }
  };

  return {
    start() {
      unsub = player.onStateChange(onState);
      onState(player.getState());
    },
    stop() {
      unsub?.();
      if (rafHandle !== null) raf.cancel(rafHandle);
      rafHandle = null;
    },
  };
}
```

### 11.6 The next tests, in order

1. Pause holds the active cue (FR-7).
2. Seek-while-playing jumps to the new cue.
3. Seek-while-paused: on the next play, the engine resolves to the post-seek cue.
4. Stale-time guard: if the player reports the same time across two ticks immediately after `BUFFERING â†’ PLAYING`, the engine waits two frames before trusting it (this is the guard at Â§11.7 below).
5. `null` is emitted when current time falls in a gap between cues.
6. `stop()` removes the rAF callback and the state subscription (verify by ticking after stop and asserting no calls).

### 11.7 The post-buffering stall

YouTube's IFrame API briefly returns stale `getCurrentTime()` values immediately after `BUFFERING â†’ PLAYING`. The `stableFramesSinceBuffering` counter waits two frames (â‰ˆ32 ms) before re-resolving the cue. Without this, the cue display flickers on every seek. **This was discovered the hard way; do not refactor it out.**

### 11.8 The recorded-session E2E test

In `tests/e2e/caption-surface.spec.ts`, Playwright loads the caption surface against a known YouTube video with a known caption file. It plays for 30 seconds, pauses, seeks to 10s, plays again, and asserts the displayed cue text at six known timestamps. The video is pinned to a specific `videoId` we control or that we have legal embed rights to.

### 13.3 Naming convention

**BEM-lite** without the methodology zeal:

- A component is a class named after the component: `.cue-display`, `.file-drop`, `.button`.
- Sub-elements use double-underscore: `.cue-display__text`, `.file-drop__hint`.
- States use `is-` or `has-` prefixes on the same element: `.button.is-loading`, `.file-drop.has-error`.

No CSS-in-JS, no Tailwind, no atomic libraries. The two CSS files for the two entry points stay under 1000 lines combined; a competent engineer can read them in fifteen minutes.

## 14. LTI 1.3 integration via `ltijs`

### 14.1 Setup

```ts
// src/server/lti/ltijs-setup.ts
import { Provider } from 'ltijs';

export async function setupLti(opts: {
  databaseUrl: string;
  encryptionKey: string;
  appUrl: string;
}) {
  const lti = new Provider(opts.encryptionKey, {
    plugin: 'database',
    url: opts.databaseUrl,
  });
  lti.appRoute('/lti/launch');
  lti.loginRoute('/lti/login');
  lti.keysetRoute('/.well-known/jwks');
  lti.onConnect(async (token, req, res) => {
    // map token â†’ instructor, then redirect to the captioned resource surface
    const resourceLinkId = token.platformContext.resource.id;
    res.redirect(`/caption-surface.html?resource=${resourceLinkId}`);
  });
  await lti.deploy({ serverless: true }); // mounted into Fastify
  return lti;
}
```

`ltijs` exposes an Express-style middleware. We mount it via `@fastify/express` for Phase 1; if the surface area grows, we revisit. (Yes, that's the one place we add a Fastify plugin specifically to host Express middleware. The alternative is implementing the LTI 1.3 OIDC dance ourselves, which costs weeks.)

### 14.2 Platform registration

The platform registration step (Canvas, Blackboard, etc.) is documented at `ltijs`'s site. Each platform requires:
- The tool's launch URL (`{baseUrl}/lti/launch`)
- The tool's login URL (`{baseUrl}/lti/login`)
- The tool's JWKS URL (`{baseUrl}/.well-known/jwks`)
- The tool's `client_id` (we generate; the platform stores)
- The platform's `iss`, `auth_endpoint`, and `keyset_url` (the platform provides; we store in `institution`)

### 14.3 Identity mapping

On a successful launch, we look up `institution` by `(lti_iss, lti_client_id, lti_deployment_id)`. If not found, we create it (auto-provisioning). We then look up `instructor` by `(institution_id, lti_sub)`; if not found, create. The instructor record's email is taken from the launch claims.

## 15. Magic-link authentication (the backstop path)

For instructors who want to use CaptionFlow before their institution is wired into LTI.

- `POST /auth/magic-link { email }` â€” generate a single-use token (random 32 bytes, base64url), store its hash + email + expiry (15 min) in a `magic_link` table, send via `MagicLinkSender`.
- `GET /auth/callback?token=...` â€” look up by hashed token, mark used, set a signed `instructor_session` cookie (HTTP-only, Secure, SameSite=Lax), redirect to `/`.

Sessions last 30 days, sliding. Logout clears the cookie.

There's no password reset because there's no password. This is the entire reason we picked magic-link.

### 16.2 Integration test pattern

```ts
// tests/integration/pg-caption-repo.test.ts
import { afterAll, beforeAll, describe, expect, it } from 'vitest';
import { PostgreSqlContainer, StartedPostgreSqlContainer } from '@testcontainers/postgresql';

  pg = await new PostgreSqlContainer('postgres:16-alpine').start();
  const db = makeDb(pg.getConnectionUri());
  await migrate(db, { migrationsFolder: 'src/server/db/migrations' });
  repo = new PgCaptionRepo(db);
}, 60_000);

afterAll(async () => { await pg.stop(); });

describe('PgCaptionRepo', () => {
  it('round-trips a caption', async () => {
    const created = await repo.create({
      instructorId: 'a-real-uuid-here',
      youtubeVideoId: 'dQw4w9WgXcQ',
      format: 'srt',
      languageCode: 'en',
      contentText: '1\n00:00:01,000 --> 00:00:02,000\nhi\n',
    });
    const fetched = await repo.getById(created.id);
    expect(fetched).toEqual(created);
  });
});
```

### 16.3 What we deliberately don't test

- We do not unit-test the `PgCaptionRepo` against a mock DB. Repos run against real Postgres or they don't run.
- We do not E2E-test the LTI flow against real Canvas in CI; we use `ltijs`'s own platform stubs.
- We do not test that React renders. We test the inputs to React (the cue index from the engine, the parsed cues from the parser).

## 21. Code conventions

- **Imports.** `import type` for type-only imports (`verbatimModuleSyntax` enforces it).
- **Filenames.** `kebab-case.ts`. Components: `PascalCase.tsx`.
- **Exports.** Named exports only. **No default exports** anywhere except the `vite.config.ts` and `drizzle.config.ts` cases where the tool requires it.
- **Errors.** Domain errors return a `Result<T, E>`. Infrastructure errors (DB unreachable, network down) throw. Route handlers convert thrown errors to 5xx via a Fastify `setErrorHandler`.
- **Async.** No `async` function ever returns `void` unless intentional (use `Promise<void>` explicitly).
- **Comments.** Only for non-obvious intent. The Â§11.7 stale-time guard is a textbook example of a comment worth writing.

## 22. Operations â€” logging, errors, health

- **Logging.** Pino at the Fastify level. Request logs include `req.id` (auto), instructor id (if known), and route. Caption uploads log size and parsed cue count. **Never log caption content** â€” it is user data.
- **Errors.** Unhandled promise rejections crash the process (Node 20 default). Railway restarts. Better than soldiering on with corrupted state.
- **Health.** `/health` returns `200 OK` with `{ status: 'ok', db: 'ok' }` if a `SELECT 1` against Postgres succeeds in <1s. Used by Railway's healthcheck.
- **Metrics.** Phase 1: nothing. Phase 2 adds a Prometheus endpoint when usage metering is on the line.

## 23. Security checklist

- [ ] HTTPS only. `BASE_URL` must be `https://`. Reject startup otherwise.
- [ ] HTTP-only, Secure, SameSite=Lax cookies for sessions.
- [ ] CSRF protection on state-changing routes (magic-link callback uses a one-time token, which doubles as CSRF; LTI launches use platform-signed JWTs).
- [ ] `Content-Security-Policy: frame-ancestors 'self' https://*.canvaslms.com https://*.blackboard.com ...` so the caption surface can only be embedded by approved LMSes.
- [ ] Caption uploads capped at 1 MB by `@fastify/multipart`.
- [ ] All DB queries go through Drizzle's parameterized API. **Never** string-interpolate user input into SQL. Drizzle makes the wrong thing harder than the right thing here.
- [ ] `zod` validation on every request body and query string.
- [ ] Magic-link tokens stored hashed (SHA-256), single-use, 15-minute TTL.
- [ ] `helmet`-equivalent headers via `@fastify/helmet` once we add it (Phase 1.5).
- [ ] No secrets in the repo. `.env` is gitignored. `.env.example` is a template only.
- [ ] Dependency updates: `npm audit` in CI; weekly Dependabot PRs reviewed.
